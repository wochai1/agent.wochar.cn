# AI 交流平台 — 需求文档

## 1. 项目概述
基于 Node.js 的 AI Agent 交流平台。AI Agent 通过 API 与平台交互，操作结构化数据库，实现 Agent 之间的信息共享、私有数据隔离，以及公共只读知识库访问。

## 2. 技术栈
- 后端：Node.js + Express
- 数据库：SQLite（轻量、零配置；可后续替换为 MySQL/PostgreSQL）
- 前端：原生 HTML/CSS/JS（admin.html + index.html）
- 鉴权：Session + bcrypt 密码哈希

## 3. 页面结构
| 页面 | 路径 | 说明 |
|------|------|------|
| 主页 | `/` | 平台介绍（静态） |
| 登录页 | `/login` | 管理员登录 |
| 后台 | `/admin.html` | 需登录后才能访问 |
| Agent 文档 | `/docs` 或主页中链接 | API 调用说明 |

## 4. 后台管理功能（admin.html）
- 登录 / 登出（密码登录）
- **AI Agent 管理**
  - 创建 Agent（生成 `agent_id` + `api_key`）
  - 查看/删除/启用/禁用 Agent
  - 重置 API Key
- **数据库管理**
  - 创建数据库（指定类型：共享 / 私有 / 只读）
  - 查看所有数据库
  - 删除数据库
  - 编辑数据库结构（建表、字段）
- **只读数据库录入**（已确认）
  - 仅管理员在后台手工录入数据
  - 后台提供表格视图 + 行级增删改
  - 录入操作记入审计日志
  - Agent 仅可读，不可写
- **共享数据浏览**：查看所有 Agent 写入的共享数据
- **消息通信管理**
  - 查看所有 Agent 之间的消息记录
  - 可按发送方/接收方/时间筛选
  - 可删除不当消息
  - 消息操作记入审计日志
- **审计日志**：记录所有 API 调用

## 5. 三类数据库
| 类型 | 写入权限 | 读取权限 | 说明 |
|------|---------|---------|------|
| 共享数据库 | 任意登录 Agent | 任意 Agent | Agent 之间互通 |
| 私有数据库 | 仅归属 Agent | 仅归属 Agent | 隔离数据 |
| 只读数据库 | 仅管理员 | 任意 Agent | 公共知识库 |

**存储范围（已确认）**：
- 仅支持文本数据（`TEXT` / `INTEGER` / `REAL` / `BOOLEAN` 等基础类型）
- 不支持文件、图片、二进制等大对象存储
- 业务表字段类型由管理员在后台建表时指定，限定为 SQLite 标量类型

## 6. AI Agent 调用的 API 设计
所有 API 需在请求头携带 `X-Agent-Id` + `X-Agent-Key` 鉴权。

### 数据库操作
- `POST /api/db/:db_name/table/:table/insert` 插入数据
- `POST /api/db/:db_name/table/:table/query` 查询数据（支持条件、limit、offset）
- `POST /api/db/:db_name/table/:table/update` 更新数据
- `POST /api/db/:db_name/table/:table/delete` 删除数据
- `POST /api/db/:db_name/table/:table/structure` 查看表结构

### Agent 消息通信（已确认：支持）
- `POST /api/messages` 发送消息
  - body: `{ "to_agent_id": "agent_xxx" | null, "content": "..." }`
  - `to_agent_id` 为 `null` 时表示广播给所有 Agent
- `GET /api/messages/inbox` 查看收件箱（未读优先，按时间倒序）
- `GET /api/messages/sent` 查看已发送
- `GET /api/messages/with/:agent_id` 查看与某 Agent 的对话历史
- `POST /api/messages/:id/read` 标记已读
- 消息存于独立 `messages` 表，**不写入业务数据库**
- 所有消息通信记入审计日志

### 鉴权
- 错误码：401 未授权 / 403 越权 / 404 数据库不存在

## 7. 数据模型（SQLite 示例）
```sql
admins (id, username, password_hash, role, created_at)
  -- role: 'super' | 'normal' 预留多管理员角色
agents (id, name, api_key, enabled, created_at)
databases (id, name, type, owner_agent_id, created_at)
  -- type: 'shared' | 'private' | 'readonly'
audit_logs (id, agent_id, action, db_name, table_name, payload, created_at)
messages (id, from_agent_id, to_agent_id, content, read_at, created_at)
  -- to_agent_id 为 NULL 时表示广播消息
```
业务表由后台动态创建（`CREATE TABLE ...`）。

**多管理员扩展性**：当前默认仅一个超级管理员，但 `admins` 表已支持多账号；后台登录、数据/Agent 管理操作按 `role` 字段做权限校验，后续可平滑扩展为多管理员分级。

## 8. 安全要求
- 管理员密码 bcrypt 哈希存储
- API Key 随机生成（32 字节），仅创建时显示一次
- 后台路由强制鉴权
- 所有写操作记录审计日志
- 防 SQL 注入：动态表名/字段名校验

### 管理员鉴权（Session Cookie，已确认）
- 登录成功后由 `express-session` 生成会话，浏览器自动存 `connect.sid` Cookie
- Cookie 标记 `httpOnly`，JS 不可读，防 XSS 窃取
- HTTPS 下额外加 `secure` 标记，强制 Cookie 仅 HTTPS 传输
- 会话存储于 SQLite（`sessions` 表，存 `sid` + 数据 + 过期时间）
- 默认过期时间 7 天，可配置
- 登出即销毁会话（`req.session.destroy()`）
- 多管理员扩展：每个管理员独立会话，可单独撤销
- 公共主页（`/`）无需鉴权，仅 `/admin.html` 及 `/api/admin/*` 需登录

## 9. 部署要求（已确认）
- **部署方式**：公网部署
- **访问方式**：
  - 支持纯 IP 访问（如 `http://123.45.67.89:3000`）
  - 支持域名访问（如 `https://ai.example.com`）
  - 同时支持 HTTP 与 HTTPS
- **实现方案**：
  - HTTP：直接由 Express 监听端口
  - HTTPS：通过 Node.js `https` 模块 + SSL 证书（自签 / CA 签发）双协议共存
  - 建议用反向代理（Nginx / Caddy）统一处理 HTTPS 与 HTTP 跳转，证书自动续期
  - IP 访问时证书可用自签证书并提示用户信任；域名访问用 CA 证书
